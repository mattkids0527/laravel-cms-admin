# decisions.md

## 這份文件做什麼

- 用來記錄此產品專案的重要決策
- 這份文件不只記結論，也要記為什麼做這個決定
- 若之後有人質疑某個方向，應優先回來查看是否已有決策紀錄

## 不知道怎麼開始時

- 可以先從「哪些事情我們已經做了選擇，而且以後很可能會被問為什麼」開始想
- 常見內容包含：
  - 架構切分方式
  - 流程設計取捨
  - 規則限制原因
  - 外部依賴選擇原因
- 若還是不確定，可先回頭看 `0_project.md`、`boundaries.md`、`rules.md`，把那些有明顯取捨的地方整理成決策

### 可直接請 AI 協助的 Prompt

> 請根據我的 `0_project.md`、`boundaries.md`、`rules.md`、`flows.md`，幫我整理出此專案目前已經做出的重要決策。每個決策請包含「決策內容」、「原因」、「影響」，並避免寫成純進度紀錄。

## 範例

### 採用單一產品專案結構

- 決策內容：使用單一產品專案作為主要開發與交接單位
- 原因：降低初期協作複雜度，避免多 repo 管理成本
- 影響：後續文件、handoff、spec 都跟著同一個專案走

### 先定義核心行為再補細部行為

- 決策內容：`behaviors.md` 先只記錄最重要、最影響狀態的行為
- 原因：避免過早把 spec 寫得過細
- 影響：後續可按需求再擴充細部行為

## 正式資料

### 權限系統採完全自建

- 決策內容：不使用 Spatie Laravel-Permission 或其他第三方權限套件，所有 RBAC 邏輯自行實作
- 原因：核心需求為「角色對後台選單的細粒度勾選」，自建邏輯更能貼合資料結構，不需為配合套件設計做妥協
- 影響：需自行設計 Gate 整合與快取策略；未來若需複雜 action-level 權限，須自行擴充

### 技術版本選定：Laravel 10 + PostgreSQL

- 決策內容：後端框架使用 Laravel 10，資料庫使用 PostgreSQL
- 原因：Laravel 10 為穩定 LTS 前版本，PostgreSQL 提供更完整的資料型別與查詢能力
- 影響：migration 使用 PostgreSQL 語法；.env 連線設定為 pgsql；部署環境需確認 PostgreSQL 服務

### 後台選單資料表命名為 admin_menus

- 決策內容：後台選單相關資料表統一以 `admin_menus` 為前綴命名，而非通用的 `menus`
- 原因：避免與前台菜單或其他系統內的 menus 資料表命名衝突，同時語意更明確
- 影響：相關資料表名稱為 `admin_menus`、`role_admin_menu_permissions`

### 後台選單採兩層結構，由 Seeder 固定

- 決策內容：admin_menus 分為「群組」（無路由，不可點）與「項目」（有路由，可點），內容由開發者在 Seeder 中定義，不提供後台 UI 管理
- 原因：CMS 框架初期選單結構穩定，UI 管理會增加複雜度，Seeder 管理對開發者更直覺
- 影響：新增選單需改 Seeder 並重新 seed；選單項目與路由名稱綁定

### 選單權限以角色為主體進行設定

- 決策內容：選單權限設定介面以角色為主，進入特定角色後勾選可見選單項目；超級管理員（is_protected=true）自動擁有所有選單權限，不需個別設定
- 原因：角色數量有限，以角色為主更易一次掌握整體授權狀況
- 影響：使用者的可見選單為其所有角色授權的聯集；middleware 需跳過超管驗證

### 帳號與角色採多對多關聯

- 決策內容：一個帳號可同時擁有多個角色，透過 `role_user` 中介表管理
- 原因：實際後台操作需求中，部分人員可能同時兼任多個職責，單一角色過於限制
- 影響：選單權限計算需合併帳號所有角色的授權集合（取聯集）
