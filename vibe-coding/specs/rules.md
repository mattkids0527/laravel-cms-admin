# rules.md

## 這份文件做什麼

- 用來定義此產品專案中的全域規則與不變條件
- 規則是無論在哪個流程、哪個角色或哪個狀態下都必須成立的限制
- 這份文件不描述流程順序，也不替代行為定義

## 不知道怎麼開始時

- 可以先從「什麼事情永遠不能錯」開始想
- 也可以先從「哪些限制不管誰操作、在哪個流程都一定成立」開始整理
- 若還是不確定，可先回頭看 `objects.md`、`states.md`、`behaviors.md`，把那些會限制建立、修改、轉移、審核、刪除的條件整理成規則

### 可直接請 AI 協助的 Prompt

> 請根據我的 `objects.md`、`states.md`、`behaviors.md`、`flows.md`，幫我整理出此專案應該寫進 `rules.md` 的全域規則。請優先列出一定成立的限制與不變條件，不要把流程步驟重寫成規則，並依「資料限制」、「狀態限制」、「權限限制」、「流程限制」分類整理。

## 範例

### 任務規則

- 任務必須有標題才能建立
- 已完成的任務不可再次指派

### 申請單規則

- 已送出的申請單不可直接回到草稿
- 已核准的申請單不可再次送審

### 權限規則

- 一般使用者不得審核申請單
- 管理者可查看所有任務

## 正式資料

### 帳號資料限制

- Email 為必填且必須在系統內唯一
- 密碼必須雜湊儲存，不可明文保存
- 帳號建立時必須至少指定一個角色

### 帳號狀態限制

- 非「啟用」狀態的帳號（待審核、停用、已刪除）不可登入
- 已刪除帳號不可被恢復、編輯或再次操作
- 帳號刪除為軟刪除，資料須保留於資料庫

### 帳號權限限制

- 系統中必須至少保留一個啟用中的超級管理員帳號
- 超級管理員不可停用或刪除自己的帳號
- 使用者不可修改自己的角色指派

### 角色資料限制

- 角色名稱必須在系統內唯一
- 系統保護角色（如超級管理員）不可被刪除或修改名稱

### 角色操作限制

- 已指派給一個或多個帳號的角色，刪除前須先解除所有帳號關聯
